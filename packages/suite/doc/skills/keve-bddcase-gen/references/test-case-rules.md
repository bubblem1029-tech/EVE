# 用例生成与校验规则

## 一、用例ID命名规范

**格式**：`[模块前缀]-[序号]`，如 `AG-01`、`SK-03`、`REG-02`

**核心规则**：用例ID 在项目内唯一，同一模块内序号递增。

**模块前缀生成规则**：

| 规则 | 说明 | 示例 |
|------|------|------|
| 从模块名提取 | 取模块核心词的前2~3个大写字母 | 智能应用 → AG, 技能 → SK |
| 避免冲突 | 同一项目内前缀不重复 | 如果"应用管理"和"应用详情"都想用 AG → 一个用 AG，另一个用 AD |
| 通用前缀 | 回归测试统一用 REG | REG-01, REG-02 |
| 最大长度 | 前缀不超过3个字符 | ✅ TM, ❌ TOOLMGMT |

**常见参考（非强制，按项目实际情况调整）**：

| 场景类型 | 常见前缀 | 说明 |
|----------|----------|------|
| 应用/智能体 | AG | Agent |
| 技能/技能卡片 | SK | Skill |
| 知识库 | KB | Knowledge Base |
| 工具管理 | TM | Tool Management |
| 交互/流程 | PI | Process Interaction |
| 布局/导航 | LY | Layout |
| 欢迎语/文案 | WW | Welcome Word |
| 回归测试 | REG | Regression（通用） |

**生成流程**：

```
1. 分析 MR/需求 → 列出涉及的功能模块
2. 为每个模块分配2~3字母前缀（避免冲突）
3. 模块内用例按序号递增（XX-01, XX-02, XX-03...）
4. 回归测试统一使用 REG 前缀
5. 最终验证：所有用例ID 在项目内唯一
```

## 二、预期结果编写规范

预期结果必须是**可验证的断言**，而非模糊描述。且**每个操作步骤必须有自己的预期结果**：

| 写法 | 评价 | 修正 |
|------|------|------|
| "筛选功能正常" | ❌ 不可验证 | → "筛选区域仅包含全部/我创建的/可编辑的三个选项" |
| "页面显示正确" | ❌ 太模糊 | → "天策门户按钮显示 disabled 状态，hover 提示文案为xxx" |
| "字数统计正确" | ❌ 不具体 | → "输入300字符后字数统计显示 300/300" |
| "弹窗验证生效" | ❌ 不可验证 | → "不填英文名称点击确认，提示'请输入技能英文名称'" |
| "进入页面" → "功能正常" | ❌ 步骤-预期不匹配 | → 步骤"进入页面" → 预期"页面加载完成，筛选区域可见"；步骤"观察筛选选项" → 预期"只显示3个选项" |

**核心原则**：预期结果要能直接映射为 Playwright 断言（`expect(xxx).toEqual(...)`, `expect(xxx).toBeDisabled()` 等），且每个步骤都有断言目标。

## 三、输出格式规范

**只输出 YAML 格式**，不再输出 Markdown 格式。

**核心规则：每一步骤必须有对应的预期结果**

| 规则 | 说明 | 反例 |
|------|------|------|
| 步骤-预期一一对应 | 每个操作步骤后面紧跟该步骤的预期结果 | ❌ "进入页面，点击按钮" → "筛选正常"（两个步骤只有一个预期） |
| 预期可验证 | 每个预期结果都能映射为具体的 Playwright 断言 | ❌ "功能正常" → ✅ "按钮选中态" |
| 不跳过关键验证 | 关键业务逻辑步骤必须有预期，不能只验证"页面可见" | ❌ 步骤"观察筛选选项" → 预期"页面可见" → ✅ 预期"只显示3个选项" |

## 四、质量校验（自检清单）

生成完成后，对照以下清单自检：

| 检查项 | 通过条件 |
|--------|----------|
| 步骤-预期一一对应 | 每个操作步骤都有对应的预期结果，不存在"无预期的步骤" |
| 用例ID覆盖度 | 每个变更的功能点都有对应的用例ID |
| 预期结果可验证性 | 每个预期结果都能映射为具体的 Playwright 断言 |
| 模块分组完整性 | 每个变更涉及的文件都被某个模块覆盖 |
| 操作步骤具体性 | 步骤描述足够具体，可以直接转化为 POM 方法调用 |
| 回归测试覆盖 | 至少有 1 条跨页面完整流程的回归测试 |
| 用例ID唯一性 | 同一模块内用例ID不重复 |
| 无降级预期 | 关键业务步骤的预期不是"页面可见"这种浅层断言 |

## 五、核心原则

**需求与代码不一致时，以代码为准，并在结论中标注差异**。

| 情况 | 处理方式 | 标注格式 |
|------|----------|----------|
| 需求文档描述了功能A，代码实现了功能A' | 测试用例以代码实现A'为准 | ⚠️ **需求偏差**: PRD描述为"A"，实际代码实现为"A'"，以代码为准 |
| 需求文档有功能B，代码中没有 | 不生成该功能的测试用例 | ⚠️ **需求未实现**: PRD描述的功能B在代码中未找到实现 |
| 代码有功能C，需求文档没提到 | 生成该功能的测试用例 | ⚠️ **代码额外实现**: 代码中存在功能C，PRD未提及 |
| 需求文档的文案与代码中的文案不同 | 测试用例使用代码中的实际文案 | ⚠️ **文案偏差**: PRD文案为"X"，代码实际文案为"Y" |
