Kent C. Dodds 的"测试奖杯"模型
前端社区更流行的其实是 Testing Trophy，因为前端 E2E 成本极高，集成测试才是 ROI 最高的层次。
              🏆
             ╱ ╲
            ╱E2E╲              ← 极少：仅冒烟 / 关键路径
           ╱━━━━━╲               3~5 
           条
          ╱       ╲
         ╱ 集成测试 ╲           ← 最多：组件集成 + API mock
        ╱ (Integration) ╲         ROI 最高、信心最强
       ╱━━━━━━━━━━━━━━━━╲
      ╱                  ╲
     ╱    单元测试          ╲     ← 补充：纯逻辑、边界条件
    ╱    (Unit)             ╲       快但不验证用户行为
   ╱━━━━━━━━━━━━━━━━━━━━━━━━╲
  ╱                          ╲
 ╱      静态分析 (Static)      ╲  ← 基座：零成本捕获大量错误
╱━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━╲

核心区别：奖杯模型中集成测试最宽（而非单元测试最宽），因为：
- 单元测试只验证"零件"，集成测试验证"装配"
- 前端 bug 多出在组件协作、API 对接、状态管理，而非纯函数逻辑
- 集成测试的信心度远高于单元测试（更接近真实使用）


当前keve-agent生成的那就应该是前端集成测试（Integration Test）

传统 E2E（后端视角）：
  用户浏览器 ──────────────────────────────→ 数据库
  ┃          前端          API         后端       ┃
  ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
  "端到端" = 从用户操作到数据库落盘，全链路

keve-agent 做的测试（前端视角）：
  用户浏览器 ─────────→ 前端 devServer
  ┃          前端           ┃
  ┗━━━━━━━━━━━━━━━━━━━━━━━━┛
  "端" = 只到前端渲染层，后端是 mock/stub
